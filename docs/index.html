Quiver

Open harness for open models

A self-evolving coding and research harness for the terminal, with a desktop app. Designed for the best open source models, with extensible tools, transparent and editable persistent and session memories.

Architecture

Global identity, per-thread memory, zero lock-in. Everything is a file.

~/.quiver/ ← Global (shared across projects) ├── core.json # Identity + user context ├── skills/ # Skills (reusable procedures) │ ├── system-prompt/SKILL.md # The system prompt (editable) │ └── cli-for-agents/SKILL.md # CLI design patterns └── projects/{name}/ ├── memory/ # Per-project memory │ ├── persona.txt # Agent behavior notes │ ├── human.txt # User details │ ├── project.json # Project context │ ├── user-preferences.md # Auto-learned preferences │ └── workspace-facts.md # Auto-learned facts └── .sessions/ # Session logs + state Cloud sync: auto-detects Google Drive, OneDrive, Dropbox, iCloud. Syncs to {cloud}/Quiver/ after every turn. No OAuth — just files in a folder.

Open Harness vs Closed Harness

Your harness, your memory, your model. Zero lock-in.

Closed Harness

  • Locked to one model provider
  • Memory is a black box
  • Context is opaque
  • No control over what the AI sees
  • Session history is trapped
  • Tools are fixed and proprietary
  • No self-improvement mechanism

Quiver (Open Harness)

  • Works with any OpenAI-compatible model
  • Memory is files you can read and edit
  • Context is transparent and shown before each prompt
  • Edit exactly what the AI consumes next run
  • Sessions are JSON files, resumable, syncable
  • Tools are TypeScript files, extensible at runtime
  • Agent can propose updates to its own prompt

Core Principles

Enforced in code, not just aspirational.

01

Your harness, your memory

Ability to manage context as you exactly intend. Memory is files you own, edit, and control.

02

Context Transparency

Show what enters the model call before each prompt — memory, skills, tools, model, tokens.

03

Explainability

Trace user prompts, resulting chain of thoughts, operations performed, and results from tool calls by the harness.

04

Provenance

Facts must come from provided context, not from training. Never fabricate file paths, function names, or APIs.

05

Read Before Write

Always read a file before modifying it. Enforced at the code level — write and edit tools are blocked if the file wasn't read first.

06

Minimal Edits

Prefer targeted edits over full rewrites. Use apply_patch for multi-file diffs.

07

Verify After Changes

Run tests after code changes. Fix failures before declaring success.

08

No Silent Actions

Every action is visible to the user. Never perform hidden tool calls.

09

Reversibility Awareness

Distinguish reversible from irreversible actions. Warn before destructive operations.

10

Self-Improvement

The agent can propose updates to its own system prompt. The user reviews, edits, or rejects — the agent never modifies the prompt directly.
